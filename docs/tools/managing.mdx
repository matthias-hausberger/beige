## Listing Installed Tools

```bash
beige tools list
```

Shows all installed tools with their descriptions and install sources:

```
[BEIGE] Installed tools:

  github
    Interact with GitHub using the GitHub CLI (gh).
    Source: npm:@matthias-hausberger/beige-toolkit@0.1.2

  chrome
    Control a Chrome browser via the chrome-devtools-mcp server.
    Source: npm:@matthias-hausberger/beige-toolkit@0.1.2
    Package: matthias-hausberger-beige-toolkit

  kv
    Simple key-value store.
    Source: ./tools/kv
```

## Updating Tools

Update a specific tool (re-fetches from its original install source):

```bash
beige tools update github
```

If the tool was installed as part of a multi-tool package, all tools from that package are updated together.

Update all installed tools:

```bash
beige tools update
```

Tools are grouped by source — each unique source is re-fetched once, even if it contains multiple tools.

## Removing Tools

```bash
beige tools remove github
```

Removes the tool from `~/.beige/tools/` and its meta file. If the tool was installed from a multi-tool package, the package directory is cleaned up automatically when the last tool from it is removed.

<Warning>
Agent configs referencing removed tools will fail validation at the next gateway start. Update your `config.json5` to remove the tool from any agent's `tools` array.
</Warning>

## Configuring Tools

### Global Config

Installed tools need no config entry — they're auto-discovered. Add a `tools.<name>` entry only when you want to pass configuration:

```json5
{
  tools: {
    github: {
      config: {
        allowedCommands: ["repo", "issue", "pr"],
        deniedCommands: ["repo"],
      },
    },
    chrome: {
      config: {
        headless: true,
        idleTimeoutMinutes: 15,
      },
    },
  },
}
```

The `config` object is passed to the tool's `createHandler(config)` at gateway startup. See each tool's documentation for available config options.

<Note>
`path` and `target` are optional when the tool is installed via `beige tools install`. The gateway resolves them automatically from the tool's `tool.json` on disk. You only need to specify `path` and `target` for tools that are NOT installed (e.g., tools referenced directly from a local project directory).
</Note>

### Per-Agent Overrides

Use `toolConfigs` on an agent to deep-merge overrides with the global tool config:

```json5
{
  tools: {
    slack: {
      config: { denyCommands: ["messages send", "messages draft"] },
    },
  },
  agents: {
    reader: {
      tools: ["slack"],
      // Uses global config as-is
    },
    writer: {
      tools: ["slack"],
      toolConfigs: {
        slack: { denyCommands: [] },  // Override: allow sending
      },
    },
  },
}
```

The merge is recursive for objects. Arrays and primitives from the override replace the base value entirely.

### Manual Tool Registration

For tools that aren't installed via `beige tools install` (e.g., tools in your project directory), specify `path` and `target` explicitly:

```json5
{
  tools: {
    "my-local-tool": {
      path: "./tools/my-local-tool",
      target: "gateway",
      config: { someSetting: true },
    },
  },
}
```

Relative paths are resolved against the config file's directory.

## Disk Layout

```
~/.beige/
  tools/                              # the gateway scans this directory
    github/                           # real dir (standalone) or symlink (from package)
    github.meta.json                  # install metadata (source, timestamp)
    chrome/
    chrome.meta.json
    kv/                               # standalone tool
    kv.meta.json
  packages/                           # intact multi-tool packages
    matthias-hausberger-beige-toolkit/
      tools/github/                   # symlinked from tools/github
      tools/chrome/
```

The `tools/` directory is the only thing the gateway loader scans. Everything in `packages/` is referenced via symlinks.
